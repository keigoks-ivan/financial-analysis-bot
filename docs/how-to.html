使用說明 — 如何呼喚 skill / SOP

Flagship 更新 · 八大非共識

四種觸發節奏：A 事件補丁 / B 季度 refresh / C 擴充 / D 重寫
docs/id/theses.html
觸發使用場景指令範例cadence
A 事件補丁 某條 thesis 被 event 打臉或強化（新財報、重大簽約、yield 翻車） 更新 flagship 第 X 條 thesis（{原因}） 2-4 週 / 次
B 季度 refresh 每 90 天主幹全面校正（對齊 Q 報季節） 季度 refresh flagship 每 90 天（主幹）
C 擴充新條 建完新 ID 有 cross-cutting insight flagship 加第 9 條（{主題}） 每 2-3 季
D 整版重寫 > 3 條 thesis 被打臉 或 A 補丁 > 2/月 持續 flagship v2.0 重寫 每 18-24 月
⚡ Status Bar 看燈號：/id/theses.html 頂部自動顯示 🟢 Fresh（<60 天）/ 🟡 Approaching（60-90 天）/ 🟠 B refresh 已到期（90-120 天）/ 🔴 Overdue（>120 天）。看到黃 / 橘 / 紅 就呼喚 B refresh。
你：（2026-07 底）「季度 refresh flagship」
Claude： 讀 pre_publish_check.md 跑 7 Gates → Gate 1 檢查 NVDA/AVGO/MU 等 Q 報 → Gate 2 掃 event thesis → 更新每條 thesis 的 consensus / ours / evidence → 更新 Status bar 日期 → v1.0.1 → v1.1.0 → push

Skill 之間的整合關係

三套 skill 如何協作
industry-analyst 產 ID → §11 列關聯個股
stock-analyst 建 DD 時自動掃 docs/id/INDEX.md → 命中 ID 則 §9/§11 引用（不重寫產業通論）

多份 ID 累積 5+ → 建 flagship（跨 ID synthesis）
flagship 的 thesis 被 event 打臉 → A 補丁；90 天 → B refresh

三套 skill 共用 docs/dd/INDEX.md、docs/id/INDEX.md 和 references/ 作為單一事實源。任何一處更新會透過索引檔自動傳導。

常見情境 / Troubleshooting

情境怎麼做
我不確定該跑哪個 skill describe 想要什麼結果即可（「我想知道 NVDA 現在該不該加倉」 → Claude 自動判斷該跑 PM / DD / ID）
Skill 執行到一半卡住 說「繼續」或「接下去」即可續跑；也可以說「換用 B 方案」讓 Claude 改路徑
數據跑出來感覺過時 說「這個數字有更新嗎」→ Claude 重跑 WebSearch；或直接呼叫 B refresh
想同時跑多個產業研究 說「批次跑 ID_XX、ID_YY、ID_ZZ」但注意品質可能略降（參考：一個一個跑最穩）
發布前要確認沒錯 skill v1.4 會自動跑 7 Gates；若想手動 peer review 說「讓我看草稿，還不要 push」
💡 最重要一句：你不需要記所有指令。描述想要的結果即可，Claude 會判斷該跑哪個 skill。這頁只是列出「最精準的指令」方便你想要精準控制時用。
