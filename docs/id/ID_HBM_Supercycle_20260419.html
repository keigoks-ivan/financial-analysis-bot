★ TSMC 官方 2026 symposium：HBM4→HBM5E 完整路線圖 + base die 事實修正

本節整合 TSMC 在 2026 北美技術論壇對外揭示的 HBM 進化軌跡。對本 ID 既有版本最重要的兩個修正是：（1）HBM4/4E 的 base die 是 N12FFC+ 邏輯製程，不是 3nm — 本 ID v1.0.1 把 HBM base die 描述為「3nm 邏輯 die」是事實錯誤；真正升到 leading-edge 邏輯（N3P）是 HBM5/5E 才開始的事；（2）HBM5 (2028) 與 HBM5E (2029) 的時程已從「預期 2029-2030」明確化為 2028 + 2029 兩階段，且單顆 CoWoS 內可容納的 HBM 顆數從 HBM4 世代的 12 顆，跨步到 HBM5 的 20 顆（2028）、HBM5E 的 24 顆（2029）。

1）base die 從 DRAM-process → N12FFC+ → N3P：兩段式跳級才是 HBM 真正的「foundry tax」

市場過去把 HBM 包裝成「memory 廠故事」，但 base die 工藝從 HBM3E（DRAM process）→ HBM4/4E（TSMC N12FFC+ 邏輯製程）→ HBM5/5E（TSMC N3P 邏輯製程）的兩段式跳級，意味著每一顆 HBM4 的 base die 都在 TSMC N12FFC+ 跑，每一顆 HBM5 的 base die 都在 TSMC N3P 跑。這對應的是 SK hynix / Samsung / Micron 三家 HBM 廠商必須把 base die 外包給 TSMC（Samsung 部分自己代工，但 NVIDIA 客戶端鎖定 TSMC base die），TSMC 等於切走 HBM 增量價值的一塊 — 而且這塊不會回流到 memory 廠。本 ID §1 thesis「HBM 邊界正從記憶體擴大到 foundry」，這次得到 TSMC 官方 roadmap 的直接驗證。

更重要的是，base die 從 N12FFC+ 跳到 N3P 是 三個世代的工藝跨越（N12FFC+ → N7 → N5 → N3P），而 HBM 從 HBM4 → HBM5 中間只隔兩代（HBM4 → HBM4E → HBM5）。換言之，TSMC 把 HBM5 的 base die 一次升到當代 leading-edge 的次世代節點（HBM5 量產 2028 時 N3P 已成熟兩年），這是為了在 base die 上整合更多控制邏輯與 PHY — HBM5 的 base die 已經接近一顆「mini SoC」，包含完整的 memory controller + interconnect controller。對 HBM 廠商而言，base die 越複雜代表 IP 設計門檻越高，HBM5 世代的 IP 集中度會比 HBM4 更高，三家中只有 SK hynix（最大 IP 投資）能 100% 自主，Samsung / Micron 對 TSMC base die IP service 的依賴會結構性上升。

2）IO 從 1024-bit (HBM3/3E) → 2048-bit (HBM4 起) — 真正的頻寬翻倍 enabler

HBM4 的官方 spec 說頻寬「per GPU 22 TB/s」，但這個數字被市場簡化為「速度提升」，忽略了真正的 enabler 是 IO bus 從 1024-bit 翻倍到 2048-bit（HBM4/4E/5/5E 都維持 2048-bit）。對製造端的意義是：每顆 HBM4 stack 的 base die 上需要 2x IO 的 routing + ESD 保護，這是為什麼 base die 必須升到 N12FFC+ 邏輯製程 — DRAM-process 的 metal stack 不夠細到塞 2048 條 IO。對封裝端的意義是：interposer 上每顆 HBM 的 routing density 翻倍 → CoWoS-L 的 RDL 層數從 6-8 層上升到 10+ 層，這直接對應 §2 的「CoWoS-L warpage」問題（為什麼 Rubin Ultra 退 4→2 die），HBM4 的 IO 翻倍與 CoWoS-L 物理上限是同一件事的兩個觀察視角。這個跨 ID 的因果鏈過去從未被市場完整描述。

3）2024 → 2029 一個 CoWoS 內 HBM 頻寬 +34X — 供給端 anchor

把 TSMC roadmap 的數字直接翻譯：2024 一個 CoWoS 內 HBM 總頻寬 = 1X（基準），2025 = 1.6X，2026 = 5.9X（HBM4 + 12 顆，IO 翻倍是主因），2027 = 8.6X（HBM4E），2028 = 21X（HBM5 + 20 顆），2029 = 34X（HBM5E + 24 顆）。5 年總頻寬 +34X是 TSMC 給市場的 supply-side anchor — 過去市場對 HBM 需求的 forecast 全都是 demand-side（Hyperscaler capex / Token throughput / GPU shipment 推算），TSMC 第一次給出供給端能 deliver 多少。如果需求端能消化（GPT-6/Llama 5/Claude N+2 等大模型訓練 + inference scaling），則 HBM 廠商 2026-2029 的 unit volume 上限可以一路追到 34X anchor；如果需求端撞牆（capex 放緩），34X 就變成 overhang，這會回到本 ID Kill Scenario 反方 1 的劇情。

💡 Insight §2.5（symposium roadmap 對 HBM thesis 的三個修正 / 強化）
1. 事實修正：HBM4/4E base die = N12FFC+，不是 3nm。本 ID 原版本把 HBM base die 描述為「3nm 邏輯 die」是錯的。真正升到 leading-edge（N3P）是 HBM5/5E 才開始；這對應 TSMC base die 業務的時點與毛利率 model 都需要重做
2. HBM5 (2028) + HBM5E (2029) 時程明確化：原 §2 寫「2029-2030」，現在已是 TSMC 官方 roadmap 的 2028 + 2029 — 比市場 forecast 提前 1-2 年。這對 SK hynix / Samsung / Micron 2027-2028 capex 規劃與 HBM5 IP 開發節奏都有直接影響
3. IO 從 1024 → 2048 是 HBM4 真正的 enabler：頻寬翻倍不是 DRAM 速度提升，而是 base die N12FFC+ 邏輯製程 + 2048-bit IO bus 同時實現的結果。這個事實重新定位「HBM 升級路徑」 — 從 memory cell scaling 變成 base die logic scaling，HBM 廠商的工藝瓶頸從「自家 1c DRAM」轉移到「TSMC base die yield」

🟡 反方 2：CXL memory pooling 跳過 HBM 信心：低
  • 事實 1：Intel、Samsung 推 CXL 2.0 / 3.0 規格
  • 事實 2：CXL pool + LPDDR5 可作為 HBM 替代（低成本、延遲較高）
  • 事實 3：2028+ AI 推論場景可能採 CXL pool + HBM hybrid
→ 若 CXL 成熟 → 部分推論工作負載遷移，HBM TAM 增速放緩
⚠ 證偽：2028 Q2 有 hyperscaler 公告 CXL 替代 HBM 部署
🟡 反方 3：Custom HBM（hyperscaler 獨家綁定）改寫份額 信心：中
  • 事實 1：Broadcom 與 Samsung / SK 討論 Google TPU 客製 HBM
  • 事實 2：若 Meta / AWS 與某一家獨家鎖，其他兩家損失
  • 事實 3：Custom HBM 利潤更高但研發成本大
→ Custom HBM 可能使市佔劇變；勝者取得結構性溢價
⚠ 證偽：2026 H2 首個 Custom HBM 正式發表
