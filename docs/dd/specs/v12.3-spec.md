# DD Schema v12.3 Upgrade Spec — 基本面深度 vs 估值簡化

> Status: **DRAFT — awaiting user confirmation**
> Author: 2026-05-14 session
> Trigger: 用戶觀察「估值佔篇幅 ~一半，但估值 alpha 低（市場 collective decision）；基本面 alpha 高（mechanism、客戶結構、競爭格局）」
> 預估規模影響：v12.2 → v12.3 總 token 持平（~94K），但內部 reshuffle ~25%

---

## 1. Mission Statement

DD 的競爭優勢在「**基本面信息深度**」（execution moat mechanism、客戶 dual-track 結構、競爭威脅分級、capital allocation 紀律），不在「**估值算法**」（Reverse DCF 隱含 CAGR、五角驗證、便宜理由檢驗、R:R 壓力測試）。後者每家 sell-side 都有同樣的數字，無 alpha。本次 v12.3 重新分配 token budget：

- 估值章節（§2 + §13）從 ~40% → **~18%**（瘦身 55%）
- 基本面章節（§5 + §8 + §9 + §11 + §12）從 ~35% → **~55%**（擴充 57%）
- 其餘（§1 + §3 + §4 + §6 + §7 + §10）~25% → ~27%（基本不變）

**核心原則**：保留估值「**訊號層**」（燈號、分位、PEG、R:R 三個數字、同業 sanity check）；砍估值「**計算層**」（Reverse DCF 反推、五角驗證、便宜理由五問、R:R 壓力測試 4 情境）。

## 9. 風險與緩解

| 風險 | 緩解 |
|:---|:---|
| 估值章節砍太多，遇到「明顯估值便宜但基本面合理」case 會少 evidence support | 保留 §2 D + §13.1 + §13.2 + §13.4 四個訊號層 + dd-meta JSON 完整欄位，足以 support 估值結論 |
| 基本面擴充會讓 LLM 寫到 token limit | v12.3 預估總 token 跟 v12.2 接近（~94K），是 reshuffle 而非膨脹。若 pilot 跑超 token，再縮 §11 三維展開 |
| 用戶 actually 想保留某些估值章節（如 Reverse DCF 喜歡看） | 本 spec 列出所有砍的具體 cite，等 review confirm 後再動 skill |
| §9 二維拆解（execution / pricing power）對某些業務模式不易區分（如 SaaS 訂閱型） | 二維可合併為「綜合分」+ 在 narrative 中說明哪個維度主導，不強制硬拆 |
| §5 5.F single thing 與 §1 trap 定性 / §5.C R1-R3 重疊 | **三者職責矩陣（強制對齊）**：(1) **§1 trap 定性** = thesis 整體是否為「價值陷阱」的**靜態裁決**（🟢 非陷阱 / 🟡 觀察期 / 🔴 高風險陷阱），時間軸 = 「現在」；(2) **§5.C R1-R3** = thesis 持有期間可能發生的**過程性風險**（margin 壓縮、客戶流失、capex bloat），時間軸分 ⚡/🔥/🐢，配「監測指標 + 警戒閾值」（continuous monitoring）；(3) **§5.F Single Thing** = **單一具體 binary trigger event**（如 "Mariana 占比 > 50%" / "Apple 切換 modem 供應商" / "FDA Phase III readout 結果"），時間軸 = 「未來某個 specific moment」（discrete event）。<br/>**寫的時候反向 check**：R1-R3 中若有 binary event 性質的 → 移到 5.F；5.F 若有 process risk 性質的 → 移到 R1-R3；§1 trap 不該重複 5.F single thing。三者各司其職、不重疊 |
