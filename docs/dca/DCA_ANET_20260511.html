§3｜Key Drivers（1-2 個命脈）

#Key Driver為什麼是命脈對應的可觀測指標

為什麼是這 2 個，而不是其他？市場常把 ANET 的命脈說成「AI Capex」 — 但這太籠統。AI Capex 是 H1 必要條件而非充分條件：如果 AI Capex 飆升但 Spectrum-X 全拿，ANET 仍會輸；如果 AI Capex 減速但 Ethernet vs InfiniBand 結構持續，ANET 仍能拿基本份額。真正決定 ANET 5Y 命運的不是「AI Capex 多大」而是「在 AI Capex 裡，ANET 拿到的軟體 + 開放生態組合是否被替代」。

⚖ Driver 抽象層 guardrail 校驗

校驗：driver 2「Ethernet vs InfiniBand 翻轉」可在 driver 1「EOS lock-in」不成立的情況下單獨被推翻嗎？✅ 可以 — 想像情境：Microsoft 部分 hyperscaler Pod 仍使用 ANET EOS（driver 1 成立），但同時整個產業 InfiniBand 因 NVDA Quantum-X800 取得 50%+ 翻盤（driver 2 被推翻），ANET 的 TAM 結構性收縮但 EOS 軟體層仍鎖定 existing customer。兩條 driver 因果鏈獨立，通過 guardrail。

護城河子來源分析

EOS 單一 OS 設計：與 Cisco IOS / NX-OS / IOS-XR 多 OS 模式不同，ANET 從 2010 年起即用單一 EOS 跨所有 7000 系列硬體。對 hyperscaler 而言，這意味著「升級新硬體不需重寫自動化腳本」。MSFT 過去十年在 Azure 部署的 ANET 設備已涵蓋多代 chipset（Tomahawk 3 → 4 → 5），但其內部 Azure Automation Tools 與 ANET API 已深度綁定 — Boris-style verify-app 模式：MSFT 內部很難自我審計「我可以放棄 ANET」的真實成本。

CloudVision 軟體層：除了 EOS 本身，ANET CloudVision 提供 telemetry + AI ops + change management 等軟體層；這是 Spectrum-X 目前缺乏的 — NVDA 強在 GPU + Switch 整合，但網路 ops 層仍需第三方工具（如 Ansible / Cisco DNA Center 或 ANET CloudVision）。短期內 Spectrum-X 只能搶硬體層，搶不到軟體層。

customer playbook lock-in：根據 ANET FY25 法說，Meta 在自家 FBOSS 平台中已嵌入 ANET ASIC 級別 telemetry hooks，這代表「Meta 自己也在 ANET 生態裡寫了大量定制邏輯」。要切換到 Spectrum-X，Meta 不只要換硬體還要重寫 FBOSS 對應層。這條 lock-in 是「結構性」而非「合約性」 — 沒有合約強迫 Meta，但工程成本實質上鎖住。
