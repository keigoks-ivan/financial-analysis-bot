## FAIL 詳情（依嚴重度排序）

### F1（blocker）— SVB 分歧錨點「+13.2」時間點與敘事錯置，且與實際 SVB 週動態相反
**位置**：`docs/monitor/index.html:344`
```
其與系統性分數的分歧（int−s）是核心讀法（2022 全年高壓時曾高出約 +15.8，2023-03 SVB 約 +13.2）
```
**查證**（對 `score_history.json` `series[]` 全序列逐日算 `int - s`）：
- `+13.2` 精確對應的日期是 **2023-04-06**，不是 2023 年 3 月（SVB 破產在 2023-03-10；文案標成「2023-03 SVB」）。
- 更關鍵：真正的 SVB 事發週（2023-03-08～03-17）`int-s` 全程只在 **+1.4 ～ +6.4** 之間，遠低於 +13.2；且從 3 月 15 日起 `int` 反而**低於** `s`（-2.5～-3.6），即恐慌當下「內部」比「系統性」更平靜——與文案想傳達的「內部長窗能捕捉到系統性分數看不到的 risk-off」敘事**方向相反**。真正達到 +13～+24 這個量級的分歧其實發生在 4 月（4/6=+13.2、4/25=+23.8），是 SVB 恐慌過後的餘波（First Republic 壓力擴散期），不是 SVB 事發當下。
- `+15.8`（2022 年錨點）本身可查證存在（精確落於 2022-10-24），且落在 2022 年 9-12 月整體 14.5～21.2 的高壓區間內，方向上站得住，但不是全年最大值（12/23 達 +21.2）——這條較輕微，可視為 cosmetic，非本項 FAIL 主因。
- 註：此 +13.2／2023-03 SVB 的措辭與 `method.internals.note_long` 資料層欄位逐字一致，可能是資料層先寫錯、HTML 只是照抄；但既然這段文字直接進了使用者可見的方法論複選框，仍需在本檔（或請上游資料層）修正。

**建議最小修正**（擇一）：
- 把日期改為與真實錨點一致的時間框架，例如「2023 年春（SVB 後餘波）約 +13.2（4 月）」，不要把事發時點（3 月）跟數值時點（4 月）混在一起；或
- 若想具體舉證「SVB 事發當週」，改引用該週真實的峰值 **+6.4**（3/8），並調整敘事為「SVB 當週分歧尚溫和，真正的內部/系統性脫鉤要到餘波期才放大」，避免暗示內部指標領先偵測到了事件本身。

### F2（blocker）— 「近窗走勢」sparkline 沿用通用漲跌配色，方向語意與壓力分數相反，今日資料已可實測出錯
**位置**：`docs/monitor/index.html:851`（呼叫 `sparkline(sparkVals)`；配色定義在既有 `sparkline()` function，約行 578-594）
```js
var lastUp = vals[vals.length - 1] >= vals[0];
var endCol = lastUp ? "var(--pos)" : "var(--neg)";
```
**查證**：`sparkline()` 是既有通用 helper，語意是「漲＝綠（好）、跌＝紅（壞）」，適合股價類序列。但 `int_s`（市場內部短窗）是**壓力分數，數值愈高＝內部愈 risk-off（愈差）**——本頁自己的 band-chip 系統（`calm`=綠／`extreme`=紅）已明文採「高＝差」配色，且本檔既有的 `scoreVeloColor()`（複合分數的動能配色，約行 761-765）**刻意不用 pos/neg**，改用 `warn`/`accent`，正是為了避免「分數走高卻被塗成正面綠色」這個陷阱。`renderScoreInternals` 卻直接借用了通用 `sparkline()`，等於繞開了本檔自己已經立下的防呆慣例。

**實測今日資料**（`score_history.json` 近 120 個交易日 `int_s`）：起點 `46.7` → 終點 `61.3`，`lastUp=true` → 端點圓點會被畫成 **綠色（var(--pos)）**。但今日的 band chip／synthesis 都同步標示「近窗＝緊張」（橘色 tense chip）。也就是說**同一個子區塊裡，橘色 tense chip 講「惡化」，旁邊的 sparkline 端點卻用綠色講「變好」**——今天的真實資料就會呈現這個矛盾，不是假設性 edge case。

**建議最小修正**：`renderScoreInternals` 不要直接呼叫通用 `sparkline()`，改寫一個內部專用版本（或加參數翻轉語意），依 `scoreVeloColor` 的邏輯配色：漲（更 risk-off）→ `var(--warn)` 或 `var(--tense)`，跌（更平靜）→ `var(--accent)` 或 `var(--pos)`，不要用「漲=pos／跌=neg」的通用二元判斷。

### F3（blocker）— 「表面平靜」硬編文案在「升溫」帶會與同句稍早的用詞自相矛盾，今日資料即可重現
**位置**：`docs/monitor/index.html:856-857`
```js
if (intSVal !== null && intSVal >= tenseBound && sVal < tenseBound) {
  synth += " · 表面平靜、內部近窗轉緊張";
}
```
**查證**：`tenseBound = sd.bands.tense[0] = 60`。這條件只排除「緊張」「極端」兩帶，卻把「升溫」帶（40-59.9）也算進「表面平靜」。今日 `s=45.2` 屬於「升溫」帶——句子最前面已經明講 `系統性 升溫`，三個詞之後卻又說 `表面平靜`，**同一句話自我矛盾**（「系統性 升溫 · 內部長線 常態 · 近窗 緊張 · 表面平靜、內部近窗轉緊張」）。這不是理論上的邊界案例，是今天實際會渲染出來的文案。

**建議最小修正**：把觸發門檻從 `tenseBound`（60）收緊為真正的「平靜／常態」上界，例如 `sd.bands.warming[0]`（40），只在 `sVal < 40` 時才用「平靜」字眼；或者乾脆別用固定詞「表面平靜」，改成動態插入 `bandS` 對應的中文標籤，例如 `"· 系統性" + BAND_LABELS[bandS] + "、內部近窗轉緊張"`，這樣永遠不會跟前面剛印出的 band 名稱衝突。
