## Batch 9 共識（Systemic Issues）

### 發現 1：章節編號不一致問題 — FORM 最嚴重，其他 ticker 標準

本 batch 中 FORM 的章節編號系統性偏移（整體 -2 對比標準 v12.3）是唯一的結構性問題。FIX / FN / FTNT 的章節號碼與標準 schema 對齊，FICO 也對齊。建議在 DD 升級流程中加一個「章節號碼對照 checklist」防止類似問題。

### 發現 2：dd-meta moat + signal 非標準組合 — FN 案例

FN 的 `moat: C` + `signal: B` 是 batch 系列中罕見的組合，代表一個特殊路徑（C 級護城河因體質全過 + ROIC 強，邊際升 B 訊號）。這個路徑未來應考慮在 dd-meta schema 加 `moat_override_reason` 欄位，讓下游 filter / 聚合（如 `collect_dca_moat_trend_map`）不被 `moat: C` 誤導。

### 發現 3：業務結構性的 top-5/10 缺口 — FICO / FIX / FTNT 三家均有

FICO（多分散 lender）/ FIX（10-K 不揭露個別客戶）/ FTNT（channel-led，終端 700K+ 分散）三家均面臨 top-5/10 具體數字缺口。正確處理方式是：
- FICO：data-gap aside 標注，✅ 已做。
- FIX：標注「分析師估算」，✅ 已做。
- FTNT：用分銷商 AR 替代，並在 data-gap 表格說明，✅ 已做。
建議統一成「業務結構說明 → 替代計量邏輯 → 來源」三段格式作為跨 ticker 標準。

### 發現 4：所有 5 檔 §13.0/13.3/13.5 均已清除 — 無殘留問題

本 batch 的一個重要 systemic fear（§13.5 body 未真正刪除）在所有 5 檔均未發現。所有報告均只保留 13.1/13.2/13.4，且無殘留 13.0/13.3/13.5 body 內容。

### 發現 5：§5.F Single Thing 全部到位，品質良莠不齊

5 檔 §5.F 全部存在，但品質有差異：
- 最佳：FIX（hyperscaler capex < +20% 精確定義，並說明與 §5.C R1 的職責分工）。
- 最佳：FN（NVIDIA CPO mix > 30% vs > 50% 兩層觸發設計，§5.F vs §9 QC-23 R2 職責清晰）。
- 合格：FICO（LLPA ≥ 10bps）、FORM（Technoprobe 認證）、FTNT（SASE billings ≥ 35%）。
均為 binary discrete event，符合 v12.3 規格。
